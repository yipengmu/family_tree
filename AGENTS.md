# Agent 项目协作指南

本文件是 Agent 在本项目中工作的执行入口。产品定位、技术边界和长期原则以 [SPEC.md](./SPEC.md) 为准；项目介绍、运行方式和参与方式以 [README.md](./README.md) 为准。

## 文档索引与阅读顺序

开始任何涉及产品、数据、权限或部署的任务前，按以下顺序阅读：

1. [README.md](./README.md)：了解项目是什么、当前已有能力、如何运行，以及用户如何参与共创。
2. [SPEC.md](./SPEC.md)：了解“看家谱、续家谱、管家谱”的产品定位、公益原则、数据模型、安全边界和演进路线。
3. 本文件：遵循代码修改、测试、安全和文档维护约束。
4. 代码、数据库 schema、迁移和测试：用当前实现验证文档描述，不把历史方案当作现状。

文档职责如下：

| 文档        | 职责                                                   | 维护时机                               |
| ----------- | ------------------------------------------------------ | -------------------------------------- |
| `README.md` | 项目入口、快速开始、目录、主要 API、当前能力           | 运行方式、对外定位或参与方式变化时     |
| `SPEC.md`   | 产品愿景、功能边界、隐私原则、架构方向、Agent 工作协议 | 持久产品规则、数据规则或系统边界变化时 |
| `AGENTS.md` | Agent 执行规范和交付要求                               | 工程协作方式或 Agent 约束变化时        |
| `docs/`     | 已废弃目录，仅保留迁移提示                             | 不新增正式文档                         |

不要在 `docs/` 下新增设计说明、故障总结、临时方案或路线图。一次性调试信息应留在代码注释、测试、提交记录或 issue 中；长期有效的规则应更新到 `SPEC.md`。

## 项目结构

- `src/`：React 单页应用；页面和组件在 `src/components/`，API 客户端在 `src/services/`，通用逻辑在 `src/utils/`。
- `api/`：生产环境的 Vercel Serverless Functions。
- `server/`：本地 Express 运行时、OCR 代理和兼容处理器；必须与 `api/` 保持 API 契约一致。
- `prisma/`：PostgreSQL schema 和数据库迁移。
- `lib/`：共享运行时代码，例如 Prisma、认证和家谱数据处理。
- `public/`：静态资源。
- `scripts/`：部署、数据库和诊断脚本。
- `tests/`：集成、端到端和调试测试。

## 产品与数据约束

本项目是公益性的多账户数字家谱，核心主张是“看家谱、续家谱、管家谱”。服务对象既包括从零开始建谱的普通家庭，也包括只有纸质册子、照片或长辈记忆的家庭。目标是让家族信息可以被长期保存、理解、补充并传给子孙，而不是只完成一次数据录入。

- 默认私密：家谱空间必须隔离，默认只允许受邀成员访问。
- 租户授权：每次租户读写都必须同时校验 `tenantId`、当前用户身份和 `TenantMembership`/角色，不能只验证 JWT 有效。
- 示范数据隔离：穆氏示范家谱只读，必须与用户自己的家谱空间分离，不能被用户写入或误当作平台公共数据。
- 数据兼容：保留现有 `FamilyData` JSON 兼容读取，规范化实体要渐进迁移；不得静默替换已经确认的事实。
- 事实可追溯：允许未知、待考、多个说法和来源并存，不为了展示整洁而擅自补全或抹平争议。
- OCR/AI 边界：AI 和 OCR 只产生草稿；保存原图、原文、结构化候选、操作者和确认状态，人工确认后才能进入正式家谱事实。
- 敏感信息保护：在世人物、未成年人、住址、联系方式、证件和身份信息默认受保护，API 输出应按查看者权限和隐私规则裁剪。
- 数据可带走：设计导出、备份和恢复能力，避免用户因平台、供应商或部署变化失去家族资料。

## 开发与验证命令

```bash
npm install --legacy-peer-deps
npx prisma generate && npx prisma db push
npm run dev                 # React :3000 + Express :3003
npm run lint
npm run format:check
npm test
npm run build
```

补充命令：

- `npm run vercel-build`：生产构建；包含 Prisma 迁移部署步骤。
- `npm run db:test`：数据库连接检查。
- `npm run test-*`：运行对应的专项服务检查。
- `.env` 从 `.env.example` 配置，永远不要把 provider 密钥放入 `REACT_APP_*` 环境变量。

根据改动范围选择验证方式。涉及前端行为时至少运行相关 lint、测试和 build；涉及认证、租户、数据模型、OCR 或隐私时，必须增加对应的专项检查。发现已有无关失败时，要区分并报告，不要通过修改无关代码掩盖。

## 编码规范

- 使用 JavaScript/JSX、两空格缩进、分号，并遵循相邻文件的引号风格。
- React 组件使用 PascalCase，例如 `FamilyTreePage.js`。
- 函数和变量使用 camelCase；组件样式放在相邻的 `.css` 文件。
- 优先小范围、可回滚的修改，避免顺手重构无关模块。
- 修改数据模型或 API 时，同时检查生产 `api/`、本地 `server/`、前端 service、Prisma schema 和相关测试。

## 测试要求

新增或修改行为时，优先补充聚焦测试，尤其是：

- 认证与租户隔离；
- Owner、Editor、Contributor、Viewer 的角色边界；
- 版本冲突和并发保存保护；
- 在世人物、未成年人和联系方式的隐私过滤；
- OCR 草稿到人工确认的流程；
- 数据导出、恢复和删除的安全性。

前端使用 CRA/Jest；后端专项检查位于 `tests/integration/`。界面变化应在本地真实路由验证，不只依赖静态代码推断。

## 安全与隐私

- 禁止提交 `.env`、JWT/数据库/OSS/邮件/OCR 凭据、生成的 dump 和真实家族隐私数据。
- 生产环境缺少 `JWT_SECRET` 时必须 fail closed，禁止使用公共 fallback secret。
- 长期密钥和审计敏感操作必须留在服务端；浏览器只能获得必要的限时签名或脱敏数据。
- 审核迁移、删除、导出、公开分享和权限变更，确认不会越过租户边界或造成不可恢复的数据丢失。

## Git 与交付

- 使用简洁的 Conventional Commit 风格主题，例如 `fix(auth): reject expired codes` 或 `feat(mobile): improve tree navigation`。
- PR 需要说明影响范围、迁移或配置变化、验证命令和 UI 截图/测试步骤；有 issue 时附上关联链接。
- 完成任务时说明：改了什么、验证了什么、哪些检查未运行及原因、是否需要同步更新 `README.md` 或 `SPEC.md`。
- 改变产品定位、权限、隐私、数据模型、API 契约或部署路径时，必须同步维护 `SPEC.md`；改变运行方式或对外入口时，必须同步维护 `README.md`。
